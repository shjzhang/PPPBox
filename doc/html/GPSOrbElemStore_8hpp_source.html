202  /*
203  * Explanation of find( ) function for GPSOrbElemStore
204  *
205  * The findOrbElem( ) funtion
206  * does the best possible job of emulating the choice
207  * that would be made by a real-time user following the
208  * selection crieteria and warnings defined in the
209  * IS-GPS-200.
210  *
211  * It is strongly suggested that the user load ALL
212  * available set of orbital elements into the store,
213  * regardless of health status. It is furthermore
214  * suggested the user call rationalize( ) to adjust
215  * the begin/end times of validity after loading
216  * all the elements and before using the store.
217  *
218  * There exists a second find fuction, findNearOrbElem( ).
219  * This is provided for compatibility with past uses
220  * of the GPSEphemerisStore class. findNearOrbElem( ) MAY
221  * return elements that are outside the range of
222  * validity and therefore need to be used with caution.
223  * Therefore,if you wish
224  * to use findNearOrbElem( ), you should do so directly,
225  * carefully examine the resulting set of orbital elements
226  * and make an informed decision before using the
227  * OrbElem.get????( ) functions.
228  */

273  /*
274  * Notes regarding the rationalize( ) function.
275  * The timing relationships defined in IS-GPS-200 20.3.4.5 mean
276  * (1.) The end of validity of a given set of orbital elements
277  * may be determined by the beginning of transmission of a new
278  * upload.
279  * (2.) The beginning of validity of the SECOND set of elements
280  * following and upload should be Toe-(0.5 fit interval) but
281  * it is not practical to differentiate between the first and
282 
283  * second set following an upload when only looking at a
284  * single set of elements.
285  *
286  * The rationalize( ) function is a means of addressing these
287  * shortcomings. The intention is to load all the navigation
288  * message data in the store, then call rationalize( ). The
289  * function will sweep through the ordered set of elements and
290  * make appropriate adjustments to beginning and end of
291  * validity values. In general, the only changes will
292  * occur in set of elements immediately before an upload,
293  * the first set following the upload, and (perhaps) the
294  * second set following the upload.
295  *
296  */
297  void rationalize( );
